# Phase 1: 가맹점/차량/기사 마스터 데이터 + 통합 검색

## 배경

Cabsy(캡시)는 콜택시 프랜차이즈 사업을 운영하며, 지금까지 가맹점(법인/개인)·차량·기사 데이터를
지역(서울/경기)×유형(법인/개인)별로 나뉜 22개의 구글시트로 관리해왔다. 여러 시트를 오가며 정보를
찾아야 해서 불편하다는 것이 가장 큰 페인포인트였다. Phase 1은 이 데이터를 하나의 DB로 통합하고,
지역/유형 관계없이 가맹점명·차량번호·기사명으로 한 번에 검색할 수 있게 한다.

이번 스프레드시트 검토로 확인한 실제 업무 흐름:
- **법인**: 회사가 계약 → 회사 소유 차량에 캡시 브랜드 작업(래핑) → 그 차량을 몰 기사를 캡시에 교육 신청
  → 캡시가 "기사 교육 수료 + 차량이 캡시 차량으로 확인됨"을 확인하면 콜 전환 완료.
- **개인**: 개인택시 기사 한 명 한 명이 각자 사업자로서 직접 계약 (8단계: 가맹신청 인입 → 정보공개서
  발송 → 교육수강 → 서류제출 → 계약서작성 → 입금 → 차량작업 → 차량작업 확인/콜전환). 이 8단계는
  Phase 2에서 다룬다.
- 법인은 가맹점 1개당 차량 여러 대, 기사 여러 명이 소속되며, **어떤 기사가 어떤 차량을 모는지 1:1
  매칭 정보가 시트에 있다** (사용자 확인).
- 개인은 가맹점=차량 1대=기사 1명이 거의 항상 성립한다 (드물게 예외 가능, 사용자 확인 — 그래도 구조는
  법인과 동일하게 3개 엔티티로 분리해 예외 케이스도 자연스럽게 수용한다).

## 범위

**포함**: 가맹점/차량/기사 마스터 데이터 스키마, 기존 구글시트(22개 사용 중 시트) 데이터 마이그레이션,
통합 검색 화면, 가맹점 상세 화면(소속 차량+기사 목록), 내부 팀 로그인.

**제외 (다음 단계)**:
- 개인 온보딩 8단계 파이프라인 UI (Phase 2)
- 콜전환/해지 전용 대시보드 (Phase 3) — 단, 콜전환 상태/일자/해지사유는 Phase 1의 기사 테이블에
  필드로는 포함한다 (데이터는 있어야 Phase 3에서 대시보드를 만들 수 있음)
- 사용자가 "지금 안 쓴다"고 확인한 시트(가맹계약 리스트, 부속합의서 전송내역, 미터점 정보, 참조 리스트
  2개, 팩스 수신여부)

## 데이터 모델

### `franchisees` (가맹점) — 공통 필드
- `id`, `region` (서울/경기), `type` (법인/개인), `business_name` (사업자명), `representative_name` (대표자명),
  `business_reg_no` (사업자등록번호, **text 타입 — 숫자로 저장 시 앞자리 0 소실 방지**), `address`,
  `phone`, `bank_name`, `account_no` (text), `account_holder`, `tax_invoice_email`, `assigned_staff`
  (담당자), `joined_at` (가맹계약체결일), `status` (활성/해지), `notes`, `created_at`, `updated_at`

### `franchisee_corporate_details` (법인 전용, franchisee_id FK, 1:1)
- 차량/배차담당자 (이름/메일/전화), 정산담당자 (이름/메일/전화), 지역본부명, 업태, 종목, 바로빌가입여부

### `franchisee_individual_details` (개인 전용, franchisee_id FK, 1:1)
- 주민등록번호 (text, 암호화 또는 최소 마스킹 처리 검토 필요 — 아래 "확인 필요" 참조), 성별, 생년월일,
  세금계산서/바로빌 역발행 이메일, 과세자지위, 역발행여부, 바로빌가입, CMS 시작월, CMS등록계좌,
  전자서명일, 정보공개서제공일, 위임장여부, 위탁여부, 개인정보동의체크

### `vehicles` (차량, franchisee_id FK)
- `plate_no` (차량번호), `car_model` (차종), `color`, `meter_point` (설치미터점), `decal_type` (데칼),
  `light_type` (갓등, 개인만 해당), `is_new_or_converted` (신규/전환), `added_at` (증차완료일),
  `terminated_at` (해지일), `status` (운행중/해지), `notes`

### `drivers` (기사, franchisee_id FK, vehicle_id FK nullable)
- `name`, `birth_date`, `gender`, `phone`, `taxi_license_no` (택시자격번호), `education_completed_at` (교육수료일),
  `call_status` (신청중/전환완료/해지), `call_converted_at` (전환일), `call_terminated_at` (해지일),
  `call_termination_reason` (해지사유), `notes`

## 마이그레이션 전략

원본 xlsx는 시트마다 헤더 위치가 다르고(대시보드용 요약 셀이 상단에 섞여 있음), 일부 셀에 깨진 수식
결과(`#REF!`, `#N/A`)가 있으며, 주민등록번호/사업자등록번호 등이 숫자로 자동 변환되어 앞자리 0이
소실된 경우가 있다.

- 시트별로 실제 헤더 행 번호를 하드코딩한 매핑 테이블을 만들어 파싱한다 (자동 감지는 이런 비정형
  데이터에서 신뢰할 수 없음).
- 주민등록번호/사업자등록번호/전화번호/계좌번호는 전부 text로 강제 변환하고, 숫자로 이미 손상된
  값(앞자리 0 소실)은 마이그레이션 로그에 남겨 수기로 검토한다.
- `#REF!`, `#N/A` 셀은 `null`로 처리하고 원본 시트명/행번호와 함께 로그에 남긴다.
- 마이그레이션은 1회성 Node 스크립트로 작성하고, 실행 로그(성공/스킵/에러 건수)를 콘솔에 출력한다.

## 인증

- 소규모 팀(2~10명), 권한 구분 불필요 — 단일 역할(staff)만 존재.
- 공개 회원가입 없음. 계정은 관리자가 스크립트로 미리 생성 (모빌잡스의 `create-admin.mjs` 패턴 재사용).
- 로그인 안 한 상태에서는 모든 페이지 접근 차단 (middleware/proxy에서 처리).

## 화면

1. **통합 검색** — 가맹점명/차량번호/기사명으로 검색, 결과에 지역·유형 뱃지 표시
2. **가맹점 상세** — 기본 정보 + 소속 차량 목록 + 소속 기사 목록 (기사별로 담당 차량 표시)
3. **가맹점/차량/기사 등록·수정 폼**

## 확인이 필요한 부분

- **주민등록번호 저장 방식**: 평문 저장은 개인정보보호법상 민감정보라 위험할 수 있다. 최소한
  DB 레벨 암호화(pgcrypto) 또는 저장 자체를 재검토하는 게 좋다 — 이 부분은 구현 전 사용자와
  다시 확인한다.
- 마이그레이션 시 `#REF!`/`#N/A`가 있던 행들은 몇 건인지 실제 스크립트 실행 후 알 수 있음 —
  수기 검토가 필요한 행 목록을 사용자에게 전달한다.
